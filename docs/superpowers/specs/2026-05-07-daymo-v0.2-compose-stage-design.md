# Daymo v0.2 — Compose Stage & Polish

**Date:** 2026-05-07
**Status:** Draft for review
**Builds on:** [Daymo v0.1 design](./2026-05-07-daymo-design.md)

## Goal

Take Daymo from "one-shot demo recorder" to "iteratable demo *editor*" by splitting the pipeline into a slow capture stage and a fast compose stage, and by adding the polish features that turn a screen recording into a produced video: scene transitions, intro and outro slates, and explicit dead-time primitives the agent can wrap around slow operations.

## Why now

After v0.1, the loop hurts. Every change — a caption typo, an overlay timing tweak, a transition idea, a different intro — re-runs the entire browser demo from scratch. That kills authoring iteration and makes Daymo feel like a one-shot tool rather than a video tool. Polish features (transitions, slates) compound this: more knobs means more re-runs.

The artifact layout already keeps the raw page video and events log next to `output.mp4`. We just don't expose them as a separable stage. v0.2 makes that split first-class.

This is also the prerequisite for v0.3's preview/studio (Remotion-style watch mode). Ship the stage split now, layer interactivity on top later.

## Non-goals

- **Voiceover / TTS.** Still deferred per the v0.1 decision. Captions stay the only narration channel.
- **Custom transition plugins.** A pluggable transition API is overengineering for v0.2. Ship a curated set; revisit if real demand appears.
- **Vertical (9:16) and square (1:1) output variants.** Currently expressible by setting `viewport` + cropping in post; a first-class flag is deferred.
- **Multi-track audio, SFX, music ducking.** Single music track behavior is unchanged.
- **Word-by-word kinetic captions, lower-thirds, picture-in-picture, B-roll inserts.** All gloss; pick up later when the iteration loop is fast enough to make experimenting with them cheap.
- **Preview server / studio.** This is the *next* spec.

## Pipeline split

v0.1 has one command:

```
daymo render <file>   →  Playwright + ffmpeg in one pass  →  output.mp4
```

v0.2 splits it into three:

```
daymo capture <file>          →  artifact bundle (page video + events + manifest)
daymo compose <bundle> [<file>]  →  artifact bundle + .demo  →  output.mp4
daymo render <file>           →  capture, then compose  (back-compat alias)
```

`compose` re-reads the `.demo` file fresh from disk. Editing prose, overlays, transitions, slates, or fx markers and re-running `compose` produces a new `output.mp4` in seconds without re-running the browser. `compose` defaults to using the `.demo` file referenced inside the bundle's manifest, so the second arg is optional when re-rendering the same demo.

### Artifact bundle layout

```
artifacts/<id>/
  capture/
    capture.json              # manifest (see below)
    events.json               # canonical events log (continuous mode)
    page.mp4                  # raw page video (continuous mode)
    scenes/                   # per-scene mode only
      00-<slug>/
        events.json
        page.mp4
      01-<slug>/
        events.json
        page.mp4
  output.mp4                  # produced by compose
  compose.log                 # ffmpeg argv + filter graph for the latest compose
```

`capture.json` manifest:

```json
{
  "version": 2,
  "demoFile": "/abs/path/to/demo.demo",
  "captureMode": "continuous",
  "viewport": { "width": 1440, "height": 900 },
  "createdAt": "2026-05-07T12:34:56Z",
  "scenes": [
    { "index": 0, "title": "Welcome", "slug": "welcome", "tStart": 0.0, "tEnd": 8.3 },
    { "index": 1, "title": "Open dialog", "slug": "open-dialog", "tStart": 8.3, "tEnd": 14.7 }
  ],
  "markers": [
    { "kind": "fastForward", "scene": 1, "tStart": 9.5, "tEnd": 12.4, "factor": 4 },
    { "kind": "skip", "scene": 1, "tStart": 13.0, "tEnd": 13.8 }
  ]
}
```

`events.json` retains the v0.1 schema, augmented with new event kinds: `sceneStart`, `fastForwardStart`, `fastForwardEnd`, `skipStart`, `skipEnd`. The manifest's `scenes` and `markers` arrays are denormalized from the events log at capture time, so the compositor reads them directly from the manifest and doesn't have to walk the full event stream to build the filter graph.

### Why a manifest separate from events.json

The events log is the chronological source of truth (overlays, fx, captions, mocks called, errors). The manifest is a denormalized index — scene boundaries and dead-time markers — that the compositor needs to build the ffmpeg filter graph. Keeping them separate keeps the events log unconcerned with compositor-specific shape.

## Capture modes

### `continuous` (default)

One browser run for the whole demo. Scene transitions in the source file are recorded as `sceneStart` events with a wall-clock timestamp; the page does not reset between scenes. This matches v0.1 behavior exactly. Compose splits the single `page.mp4` at scene timestamps and applies transitions between segments.

### `per-scene` (opt-in)

One browser run *per scene*. Each scene starts fresh from `url` (or a scene-level `url:` override), with mocks and auth re-applied. Captures land in `capture/scenes/<index>-<slug>/`. Compose concatenates the scene videos with transitions between them.

Per-scene mode unlocks `daymo capture <file> --scene <n>` (re-shoot a single scene without redoing others). It also lets a scene have its own starting URL or its own mock overrides — useful when scenes traverse logically separate pages.

Set in frontmatter:

```yaml
captureMode: continuous   # default
# or
captureMode: per-scene
```

In `per-scene` mode, scenes can declare scene-level overrides:

```markdown
# Settings page

```scene-config
url: http://localhost:3000/settings
```
```

(Only legal in per-scene mode. Parser rejects `scene-config` blocks in continuous mode with a clear error.)

## Transitions

Five types: `crossfade` (default), `dip-to-black`, `slide-left`, `slide-right`, `none`.

### Defaults

```yaml
defaultTransition: crossfade
transitionDuration: 0.5s
```

Both optional. Built-in defaults: `crossfade`, `0.5s`.

### Per-scene override

A `transition` fenced block at the top of a scene overrides the default for the transition *into* that scene:

````markdown
# Open the new-project dialog

```transition
type: dip-to-black
duration: 0.8s
```

```playwright
...
```
````

### Implementation

| Type | ffmpeg implementation |
|---|---|
| `crossfade` | `xfade=transition=fade:duration=D:offset=O` |
| `dip-to-black` | `fade=t=out` on outgoing + `fade=t=in` on incoming, with brief black hold |
| `slide-left` / `slide-right` | `xfade=transition=slideleft` / `slideright` |
| `none` | concat without transition |

Transitions before the first scene (i.e., into the first scene from the intro) and after the last scene (out to the outro) follow the same rules.

## Intro & outro slates

Both default-on, both auto-generated from frontmatter `title` and `description`, both rendered at compose time by a small headless Chromium pass against a built-in HTML template. The user's frontend is *not* loaded for slates.

### Default templates

- **Intro** (2.5s): centered title (large), description below (smaller), background `#0a0a0a`, accent stripe `#3b82f6`. Fade in (0.4s), hold, crossfade into the first scene's transition.
- **Outro** (2.0s): centered title, "Made with Daymo" footer, same background. Crossfade in from the last scene, fade out.

### Overrides

```yaml
intro:
  duration: 3s
  background: "#0a0a0a"
  accent: "#3b82f6"
  logo: ./logo.svg          # path relative to .demo file
  title: "Custom intro title"   # falls back to frontmatter title
  subtitle: "Custom subtitle"   # falls back to frontmatter description
outro:
  duration: 2s
  text: "Try it free at example.com"
  logo: ./logo.svg
```

Set `intro: false` or `outro: false` to disable.

### Why headless Chromium and not pure ffmpeg drawtext

Because the rest of Daymo's overlay system already lives in a browser. Slates re-use the same caption banner, callout, and highlight CSS that scenes use, so visual consistency is automatic. ffmpeg `drawtext` would mean a parallel typography path. The cost is one extra ~3-second Chromium pass at compose time per slate (intro + outro), which is dwarfed by ffmpeg encode time.

The slate renderer is its own module with one job: take a slate config, produce an `.mp4` at the demo's viewport dimensions and frame rate. The compositor concatenates these with the scene videos.

## Dead-time primitives

Two new fx primitives. The agent decides where to wrap.

```ts
fx.fastForward<T>(fn: () => Promise<T>, factor?: number): Promise<T>
fx.skip<T>(fn: () => Promise<T>): Promise<T>
```

`fastForward(fn, factor)`: runs `fn` at real time during capture. The compositor speeds up the captured video segment between `fastForwardStart` and `fastForwardEnd` events by `factor` (default 3). `factor` is clamped to `[1.5, 16]`.

`skip(fn)`: runs `fn` at real time during capture. The compositor cuts the segment between `skipStart` and `skipEnd` from the final video.

Both wrap a callback rather than mark a region with start/end calls — the wrap form guarantees balanced start/end events even if the callback throws (the marker is emitted in `finally`), and reads naturally:

```ts
await fx.fastForward(async () => {
  await page.waitForResponse('/api/slow');
  await page.waitForSelector('[data-loaded]');
}, 4);

await fx.skip(async () => {
  await page.waitForTimeout(2000);
});
```

### Implementation

In capture: emit `fastForwardStart` / `fastForwardEnd` (or `skipStart` / `skipEnd`) events with timestamps. The events go into the manifest's `markers` array.

In compose: build an ffmpeg filter graph that:

- Splits each scene's video segment into sub-segments at marker boundaries.
- Applies `setpts=PTS/factor` to fast-forwarded sub-segments (audio is unaffected — the only audio is the user's music track laid over the whole thing).
- Drops skipped sub-segments entirely.
- Concatenates the sub-segments back together before applying scene transitions.

Markers cannot overlap; the parser/runner rejects nested `fastForward` and `skip` calls with a clear error.

## File format changes

New optional frontmatter keys:

| Key | Type | Default | Description |
|---|---|---|---|
| `defaultTransition` | enum | `crossfade` | One of `crossfade`, `dip-to-black`, `slide-left`, `slide-right`, `none` |
| `transitionDuration` | duration | `0.5s` | Default transition duration |
| `intro` | object \| `false` | auto | Intro slate config or `false` to disable |
| `outro` | object \| `false` | auto | Outro slate config or `false` to disable |
| `captureMode` | enum | `continuous` | `continuous` or `per-scene` |

New fenced block kinds inside scenes:

| Block | Allowed in | Purpose |
|---|---|---|
| `transition` | any scene (incl. first) | Override the transition into this scene |
| `scene-config` | per-scene mode only | Scene-level `url`, `mocks`, `auth` overrides |

Existing blocks (`playwright`, `overlay`) are unchanged.

## Backwards compatibility

v0.1 `.demo` files parse without modification. They will render *slightly differently* on v0.2:

- A 2.5s intro slate prepended (auto-generated from `title` + `description`)
- 0.5s crossfades between scenes
- A 2.0s outro slate appended

This is documented as a visual change in the v0.2 release notes. To reproduce v0.1 output exactly:

```yaml
defaultTransition: none
intro: false
outro: false
```

The pipeline split is fully transparent: `daymo render` continues to work identically and is implemented as `capture` followed by `compose`.

## Components affected

| File | Change |
|---|---|
| `parser.ts` | Accept new frontmatter keys; parse `transition` and `scene-config` fenced blocks; reject scene-config in continuous mode |
| `types.ts` | New types for transitions, slates, capture mode, markers, manifest |
| `fx.ts` | Add `fastForward` and `skip`; emit balanced start/end events even on throw |
| `controller.ts` | Emit `sceneStart`, `fastForwardStart/End`, `skipStart/End` events; honor scene-config in per-scene mode |
| `runner.ts` | Split into capture orchestrator (continuous + per-scene paths); write `capture.json` manifest |
| `compositor.ts` | Read manifest; build filter graph for transitions, slates, fastForward, skip; orchestrate slate rendering pass |
| New: `slates.ts` | Headless Chromium pass that renders intro/outro slates from a built-in HTML template |
| New: `commands/capture.ts` | `daymo capture <file>` |
| New: `commands/compose.ts` | `daymo compose <bundle> [<file>]` |
| `commands/render.ts` | Becomes capture-then-compose |
| `cli.ts` | Register the two new commands |

## Testing strategy

- **Unit (`parser`)** — new frontmatter keys parse correctly; `transition` blocks parse; `scene-config` rejected in continuous mode; unknown frontmatter keys still rejected.
- **Unit (`fx`)** — `fastForward` and `skip` emit balanced start/end events on success and on throw; nested calls rejected.
- **Unit (`compositor`)** — filter graph builder produces the expected argv for representative cases: continuous + crossfade, per-scene + dip-to-black, fastForward inside a scene, skip across a scene boundary, slate concat.
- **Unit (`slates`)** — slate config → expected HTML; renderer produces an mp4 at viewport dims and frame rate.
- **Integration (`capture` → `compose`)** — capture once, compose twice with no `.demo` change, output is byte-equal. Capture once, edit captions, compose, output differs from baseline only in the captioned regions.
- **Integration (per-scene re-shoot)** — capture in per-scene mode, run `--scene 1` again, scene 1's directory updates, scenes 0/2 are untouched, compose succeeds.
- **E2E** — extend the existing sample-app fixture with a new `.demo` that exercises `defaultTransition: dip-to-black`, an `intro:` override, and a `fx.fastForward` block. Smoke check: file exists and is non-trivially sized.

## Worked example

````markdown
---
title: Create your first project
description: Walks a new user through creating their first project.
url: http://localhost:3000
viewport: { width: 1440, height: 900 }
defaultTransition: crossfade
transitionDuration: 0.5s
intro:
  duration: 2.5s
  accent: "#3b82f6"
outro:
  duration: 2s
  text: "Try it free at daymo.dev"
captureMode: continuous
music: gentle-corporate.mp3
mocks:
  - source: inline
    routes:
      "GET /api/me": { "name": "Alex", "plan": "free" }
      "GET /api/projects": []
---

# Welcome to your dashboard

Welcome back, Alex. This is your project home base.

```playwright
await page.waitForSelector("h1:has-text('Projects')");
```

---

# Open the new-project dialog

```transition
type: dip-to-black
duration: 0.8s
```

Click **New Project** to open the creation dialog.

```playwright
await fx.cursorTo("[data-testid='new-project-btn']");
await fx.fastForward(async () => {
  await page.click("[data-testid='new-project-btn']");
  await page.waitForResponse('/api/projects/new-form');
}, 3);
await page.waitForSelector("[role='dialog']");
```
````

Renders as: 2.5s intro slate (title + description, blue accent) → 0.5s crossfade → first scene at real time → 0.8s dip-to-black → second scene with the click → page wait sped up 3× → real-time at the dialog → 0.5s crossfade → 2s outro slate.

## Implementation phases

Each phase is independently testable and shippable.

1. **Parser & types.** New frontmatter keys, `transition` and `scene-config` fenced blocks, full schema validation. Tests pass with no runtime change.
2. **fx primitives.** `fastForward` / `skip` with balanced events. New events appear in the events log. Existing demos still produce the v0.1 output (compose ignores the new events for now).
3. **Pipeline split.** `daymo capture` and `daymo compose` exist; `daymo render` calls them in sequence; manifest written and read; compose still produces exactly v0.1's output. This is the structural change with no user-visible polish yet.
4. **Transitions.** Compositor consumes scene boundaries and per-scene transitions. v0.1 demos start picking up default crossfades.
5. **Intro & outro slates.** Slate renderer module, default template, frontmatter overrides. Demos now have intros/outros.
6. **Dead-time application.** Compositor consumes `markers` from the manifest, applies setpts/trim. `fx.fastForward` / `fx.skip` become end-to-end functional.
7. **Per-scene capture mode.** Runner gains the per-scene path; bundle layout extended; `--scene <n>` re-shoot. Compose handles concat-with-transitions for per-scene bundles.
8. **Tests & docs.** Backfill any missing tests, write the v0.2 README section, document the back-compat behavior change.

Phases 4 and 5 are user-visible wins. Phase 3 is the leverage move. Phase 7 can land last without blocking the rest.
